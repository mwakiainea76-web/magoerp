    SETUP -> INVOICING -> PAYMENTS -> ADJUSTMENTS -> REFUNDS -> REPORTS


1. SETUP PHASE (Configuration)
-----------------------------

FeeTemplate (e.g. "BSc Computer Science - Year 1")
  └── FeeTemplateItems (Tuition: 200,000 | Library: 5,000 | ICT: 10,000 | Exam: 8,000)
        └── CurriculumFeeAssignment (links template -> course_curriculum + academic_session + year_level)
              ├── issuance_type: per_session | per_year
              ├── year_level: 1-4 (which year of study)
              ├── session_number: 1-3 (which session within the year)
              ├── dormancy rules for fee auto-generation
              └── split_amount / split_ratio for per-session allocation

Key pages: FeeTemplatesPage, FeeTemplateItemsPage, FeeTemplateAssignmentsPage


2. INVOICE CREATION (Debit)
---------------------------

Trigger:  Admin clicks "Issue Fee" in Billing or auto-generated on enrolment
Service:  BillingService::createInvoiceForStudent()

Flow:
  Student's active CourseEnrolment
    -> resolve year_of_study + session_number
    -> find approved CurriculumFeeAssignment matching (curriculum + session + year)
    -> CREATE Invoice (status='issued', due_date = +30 days)
      -> CREATE InvoiceLineItems (snapshot of fee template items with amounts)
    -> POST StudentLedgerEntry (type='invoice', debit=amount, credit=0)
    -> APPLY any existing unallocated credits to this invoice (FIFO)
    -> SYNC StudentAccountBalance
    -> AUDIT LOG

Invoice statuses:  issued -> partial -> paid | cancelled | reversed

Billing actions that create debit entries:
  - Issue Fee      POST /invoice-adjustments    Invoice from a fee template
  - Issue Penalty  POST /invoice-charges        Standalone penalty invoice


3. PAYMENTS (Credit)
--------------------

Trigger:  Admin clicks "Record Payment" in Billing
Service:  BillingService::recordStudentPayment()

Flow:
  Student + Amount + Method (M-Pesa / Bank / Cash / Cheque / Airtel)
    -> Idempotency check (pay:{student}:{reference})
    -> CREATE Payment (status='completed')
    -> Find ALL outstanding invoices, sorted oldest-first
      -> For each invoice (FIFO allocation):
        -> CREATE InvoicePaymentAllocation min(remaining, outstanding)
        -> POST StudentLedgerEntry (type='payment', debit=0, credit=allocated)
        -> recalculateTotals() on invoice
    -> If excess remains after all invoices paid:
      -> POST unallocated credit ledger entry (credit=remaining, no invoice_id)
    -> SYNC StudentAccountBalance

Key insight: Unallocated credits are automatically applied to new invoices when created.


4. ADJUSTMENTS (Credit or Debit)
--------------------------------

Trigger:  Admin clicks "Discount/Waiver" or "Invoice Reversal" in Billing
Service:  BillingService::applyAdjustment()

Two types:
  CREDIT adjustments (reduce debt):  discount, waiver, bursary, helb, reversal
  DEBIT adjustments (increase debt):  penalty

Flow:
  Invoice + Type + Amount + Description
    -> Credit adjustments capped at outstanding balance
    -> Percentage discount computed from invoice line items total
    -> CREATE StudentFeeAdjustment
    -> POST StudentLedgerEntry (type='adjustment', debit/credit as appropriate)
    -> recalculateTotals() on invoice (recomputes balance_due)
    -> SYNC StudentAccountBalance


5. REFUNDS
----------

Trigger:  Admin clicks "Process Refund" in Billing (graduated students only)
Service:  BillingService::processRefund()

Flow:
  Student must be status = graduated
  Available credit from StudentAccountBalance (negative balance means credit)
    -> Capped at available credit
    -> CREATE Refund (status='processed')
    -> POST StudentLedgerEntry (type='refund', debit=amount, credit=0)
    -> SYNC StudentAccountBalance


6. THE LEDGER (Source of Truth)
-------------------------------

StudentLedgerEntry (per-student, per-transaction)
  type:          'invoice' | 'payment' | 'adjustment' | 'refund'
  debit (+):     money the student owes
  credit (-):    money the student paid / was given
  links to:      invoice_id, payment_id, adjustment_id (fully traceable)
  transaction_date:  chronological ordering

Running balance formula:
  balance = SUM(debits) - SUM(credits)
  Positive = student owes money (debtor)
  Negative = student has credit (overpaid)

StudentAccountBalance (cached snapshot):
  balance = total_invoiced - total_paid + total_refunded - adjustment_net
    total_invoiced   = SUM of invoice debits
    total_paid       = SUM of payment credits
    total_refunded   = SUM of refund debits
    adjustment_net   = credit_adjustments - debit_adjustments


7. REPORTS
----------

Report Types (FinanceReportsController):

  Debtors          Students with outstanding balance > 0, grouped by student + session
  Credits          Students with negative ledger balance (credit balance)
  Aging            Invoices with balance_due > 0, bucketed:
                     Current | 1-30 | 31-60 | 61-90 | 90+ days overdue
  Collections      Revenue grouped by fee template + invoice type
  Adjustments      All fee adjustments filtered by type + date range
  Penalties        Penalty-type invoices
  Refunds          All processed refunds
  Assignment Audits  Fee assignment dormant portion edit history

Dashboard Metrics (FinanceReportsDashboardController):

  Total Collected     sum of all payments
  Outstanding         sum of all unpaid invoice balances
  Collection Rate     collected / (collected + outstanding)
  Total Adjustments   sum of adjustment amounts
  Total Refunds       sum of refund amounts
  Monthly Revenue Trend  12-month bar chart data
  Top Defaulters      students with highest outstanding

API Export Endpoints:
  GET /finance/reports/export          CSV export of any report type
  GET /invoices/export                 CSV/Excel/PDF of invoices
  GET /payments/export                 CSV/Excel/PDF of payments
  GET /ledger/export                   CSV/Excel/PDF of ledger
  GET /finance/dashboard/export        CSV export of dashboard


8. FINANCIAL STATEMENT
----------------------

Sessions scope:  session_to_date | per_session | per_year | custom
Query all StudentLedgerEntry within scope
  -> Group by academic session
  -> Calculate running balance per transaction
  -> Include dormant future fees (unissued per_year assignments)
  -> Summary: total_debit, total_credit, ledger_balance

Admin:  GET /students/{student}/financial-statement
        GET /students/{student}/financial-statement/download  (PDF)
Student: GET /my/financial-statement
         GET /my/financial-statement/download  (PDF)


9. RECONCILIATION
-----------------

POST /finance/reconcile

  For all active students in a session:
    1. recalculateTotals() on each non-cancelled invoice
    2. Sync StudentAccountBalance from scratch

  Fixes any inconsistencies caused by manual DB changes or race conditions.


10. DATA MODEL (Tables)
-----------------------

fee_templates                     Master fee template definitions
fee_template_items                Individual line items within a fee template
curriculum_fee_assignments        Links fee templates to course curricula + session
invoices                          Core invoice records
invoice_line_items                Line items on each invoice (with snapshot data)
payments                          Student payment records
invoice_payment_allocations       Links payments to invoices
student_fee_adjustments           Discounts, waivers, reversals, penalties
refunds                           Student refund records
student_ledger_entries            Double-entry ledger of ALL financial transactions
student_account_balances          Cached balance snapshot per student per session
finance_audit_logs                Audit trail for finance operations
