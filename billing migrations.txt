Critical distinction: A Fee Template is never an invoice. It is a reusable fee configuration that exists before any student is billed. The word invoice must only appear once a specific student owes money.

Established Business Rules
These are confirmed and must not be broken:
Fee Templates
- Configuration only — never an invoice
- is_issued flips to true on first student billing — permanently locks
  the template and all its items as read-only
- The lock is the audit mechanism — no snapshotting needed
- Changing fees = create a new template, never edit an issued one
- Types:
    fees          → semester charges, goes through curriculum_fee_assignments
    hostel        → issued on bed allocation
    exam          → special or retake exam charges
    penalty       → late payment, disciplinary
    fine          → library, damage, other
    miscellaneous → any other one-off charge
Curriculum Fee Assignments
- Maps fee template → course curriculum + academic session + year level + session number
- Must be approved before invoice generation is permitted
- UNIQUE (course_curriculum_id, academic_session_id, year_level, session_number)
  prevents double-billing
- Only structured semester fees go through this table
- Hostel, penalty, exam, fine templates skip this table
  and are issued directly to the student
Invoices
- Student-facing billing document, generated from a fee template
- invoice_line_items are copied from locked fee template items at generation time
- invoice_template_id stored on invoice — every invoice traces to its template
- invoice_type inherited from fee_template.type
- Status lifecycle: issued → partial → paid (also waived, cancelled)
- No orphan invoices — every invoice must originate from a fee template
Payments
- student_id is never nullable — every payment belongs to a student
- method:      KCB | MPESA | HELB | CDF | COUNTY | SCHOLARSHIP | EMPLOYER | NATIONAL_BANK
- reference:   bank/provider transaction ref  e.g. 184812, H100500
- description: full human-readable source shown on fee statement
               e.g. "HELB LOAN, REC NO 321648; BATCH NO 002833"
- idempotency_key prevents duplicate recording on retry
FIFO Payment Allocation
- BillingService receives payment amount
- Automatically allocates to outstanding invoices oldest first (issue_date ASC)
- Student never chooses which invoice to pay
- One payment → many invoices
- One invoice → many payments
- Overpayment sits as unallocated credit on the ledger
- Consumed automatically when next invoice is issued to that student
- invoice_payment_allocations managed entirely by BillingService
Student Fee Adjustments
- Corrections and reversals to existing invoices only
- type: debit (additional charge) | credit (waiver, reversal)
- reference stored e.g. SIA114077 — shown on fee statement
- Must never modify an existing invoice — new financial event only
- soft_deletes — adjustments must never disappear from audit trail
- New charges (hostel, penalty, exam) are invoices, NOT adjustments
Student Ledger Entries
- Immutable financial source of truth
- Every financial event writes exactly one entry — never edited, never deleted
- No soft_deletes — the ledger cannot have hidden entries
- Corrections via new reversal entries only
- type: invoice (debit) | payment (credit) | adjustment (debit or credit)
- reference copied from source:
    invoice_number              when type = invoice
    payments.reference          when type = payment
    adjustments.reference       when type = adjustment
- description = human-readable text for Description column on fee statement
- Running balance = SUM(credit) - SUM(debit) ordered by transaction_date
- Grouped by academic_session_id for semester sections on fee statement
Student Account Balances
- Cached summary of ledger per student per session
- Never the source of truth
- Always recomputable from student_ledger_entries
- Reconciled nightly against the ledger
Who Hits What
"What do I owe this semester?"  → invoices + invoice_line_items
"Show my fee statement"         → student_ledger_entries
"What is my balance?"           → student_account_balances (cache)
"Make a payment"                → BillingService → payments → FIFO → invoice_payment_allocations
"Finance report"                → student_ledger_entries
