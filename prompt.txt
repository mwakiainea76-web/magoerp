# Implementation Prompt: Fix Ledger Debit/Credit Directions + Add Payment Reversal

## Context for the assistant

We're in **dev mode** — the schema has not shipped to production and there is no real data to preserve.
**Edit the existing migration file directly** (`database/migrations/..._create_finance_tables.php` or
whatever it's named in this repo) rather than creating a new migration, for every change listed below.
The only exception: if a change is clearly a distinct, independently-toggleable feature addition rather
than a correction to the existing tables, a new migration is fine — but default to editing in place.

This prompt fixes an accounting-direction bug found during review, and adds a Payment Reversal feature
that was previously scoped but never implemented (see the "Reverse Payment" spec from an earlier prompt
in this repo/thread — implement it now, it was accidentally dropped).

---

## 1. Fix `refunds` direction

**Bug:** Refunds were being treated as a credit to the student ledger. They must be a **debit**.

Reasoning: `balance = SUM(debit) − SUM(credit)`. A refund pays out an existing credit balance (negative
balance) back to the student in cash. To move balance back toward zero, the entry must be a debit, not
a credit. Posting it as a credit doubles the credit balance instead of clearing it, and breaks bank
reconciliation immediately.

Action:
- In `BillingService::processRefund()`, ensure the posted `StudentLedgerEntry` is
  `type='refund', debit=amount, credit=0` (fix if currently reversed).
- Update any report/query currently summing refunds as a credit (Dashboard `Total Refunds`,
  `StudentAccountBalance` sync logic) to treat refund as a debit consistently.

---

## 2. Split `reversal` into `invoice_reversal` and `payment_reversal`

**Bug:** A single generic `reversal` type is ambiguous — it can mean "we waived a charge" (credit,
no cash moved) or "we're undoing a payment" (debit, previously-received cash is being retracted). These
are opposite-sign operations and must be distinguishable in the `type` column, not left to free-text
`description` parsing.

| type | direction | meaning | FK |
|---|---|---|---|
| `invoice_reversal` | credit | cancel/waive a charge on an invoice | `invoice_id` |
| `payment_reversal` | debit | undo a payment (bounced cheque, wrong student, invalid transaction) | `payment_id` |

Action:
- Rename any existing `applyAdjustment()` reversal path to post `type='invoice_reversal'` explicitly.
- Implement `payment_reversal` as part of the new Payment Reversal feature below.

---

## 3. Add `refund_id` FK to `student_ledger_entries`

**Gap:** the ledger links `invoice_id` and `payment_id` but has no column tracing a refund entry back
to its `refunds` row — only free-text `reference`. Edit the existing migration's
`student_ledger_entries` table definition to add:

```php
$table->foreignUuid('refund_id')->nullable()->constrained('refunds')->nullOnDelete();
```

Place it alongside the existing `invoice_id`/`payment_id` columns. Update `processRefund()` to populate
it when posting the ledger entry.

---

## 4. Enforce single-sided ledger rows at the DB level

Edit the existing migration to add a check constraint (guard for sqlite the same way the existing
`invoice_payment_allocations_amount_check` constraint does):

```php
if (DB::getDriverName() !== 'sqlite') {
    DB::statement('ALTER TABLE student_ledger_entries ADD CONSTRAINT ledger_single_sided_check
        CHECK ((debit > 0 AND credit = 0) OR (credit > 0 AND debit = 0))');
}
```

## 5. Payment Reversal feature — schema

**This was scoped previously but never implemented — add it now.** Edit the existing migration's
`payments` table definition to add:

```php
$table->timestamp('reversed_at')->nullable()->after('status');
$table->uuid('reversed_by')->nullable()->after('reversed_at');
$table->string('reversal_reason', 255)->nullable()->after('reversed_by');
```

`status` column gains a new valid value: `'reversed'` (alongside existing `completed`).

## 6. Payment Reversal feature — service method

Add `BillingService::reversePayment(Payment $payment, string $reason, User $actor)`:

```
1. Guard: payment.status === 'completed' (reject if already reversed)
2. Find all InvoicePaymentAllocation rows for this payment
   -> for each: increase the invoice's balance_due back by the allocated amount
      (mark the allocation reversed for audit trail; do not delete it)
   -> recalculateTotals() on each affected invoice (status may revert from
      'paid'/'partial' back to 'issued'/'partial')
3. If this payment created unallocated credit, reverse that portion too
4. POST StudentLedgerEntry:
     type='payment_reversal', payment_id=payment.id,
     debit=payment.amount, credit=0, description=reason
   -- new row, does NOT edit or delete the original payment ledger entry
5. UPDATE Payment: status='reversed', reversed_at=now(), reversed_by=actor.id,
     reversal_reason=reason
6. SYNC StudentAccountBalance
7. AUDIT LOG
```

Wrap the whole method in a DB transaction.

## 7. Payment Reversal feature — UI on the action route page

Add the "Reverse Payment" action to the Student Finance Profile action bar (alongside the existing/
planned "Issue Invoice" and "Record Payment" actions):

```
[ + Issue Invoice ]   [ + Record Payment ]   [ ⟲ Reverse Payment ]
```

- `Reverse Payment` is only enabled if the student has at least one payment with `status='completed'`.

  - Submit button disabled on click; request carries an idempotency key.
- On the Payments table, extend the `STATUS` badge set to include `Reversed` (gray pill) alongside the

## 8. Reconciliation checks to add (scheduled job or admin-triggered)

- `student_account_balances.balance` == recomputed `SUM(debit) − SUM(credit)` per student/session from
  `student_ledger_entries` — flag drift.
- `SUM(debit WHERE type='invoice')` == `SUM(invoices.amount_due WHERE status != 'cancelled')`
- `SUM(credit WHERE type='payment')` == `SUM(payments.amount WHERE status='completed')`
- `SUM(debit WHERE type='refund')` == `SUM(refunds.amount WHERE status='processed')`
- `SUM(debit WHERE type='payment_reversal')` == `SUM(payments.amount WHERE status='reversed')`
- Every ledger row has exactly one non-null FK matching its `type` (e.g. `type='payment'` must have
  `payment_id` set and `invoice_id`/`refund_id` null) — flag any row that doesn't.
