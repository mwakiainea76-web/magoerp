I think the cleanest answer is: transcript should be anchored on `session enrolment progression`, not just `year + module`, and `module` should become optional metadata rather than the primary selector.

Why:
- `AcademicSessionEnrolment` already stores the student’s progression truth: `year_of_study` and `session_number`.
- `StudentUnitRegistration` tells us which units the student actually took in that progression.
- `Unit.modules_taught` is useful, but it is not universal. Some certification-body units span the whole year and have `modules_taught = null`, so making module mandatory breaks that model.

For the non-modular certification-body case, I would not generate transcript “per enrolled session only” in the narrow sense. I would do this instead:
- Primary transcript selector: `session enrolment`
- Transcript result: all registered + published units up to that selected progression, or exactly within that selected progression depending on the document type
- `module` only refines the result when the units are modularized

That gives us two clear transcript modes:
1. `Progress Transcript`
- Based on a selected session enrolment record
- Shows units tied to that progression point
- If units are non-modular and span the year, include the units registered under that session progression without forcing a module filter

2. `Cumulative Transcript`
- Based on the latest or selected session enrolment
- Shows all published transcriptable units up to that point

For your student `View Marks` filters, I’d refactor like this:
- `Session` as the first selector
  - values come from the student’s own `AcademicSessionEnrolment` rows
  - label example: `Year 1 Session 1 - Semester 1 2025/2026`
- `Module` as the second selector
  - populated from units registered within the selected session progression
  - if no modularized units exist, either hide it or show `All modules`
- Remove separate `Year of Study` filter from transcript if `Session` already carries progression
- For marksheet, same idea can work, but only if you want one consistent student UX

So the rule becomes:
- Admin/staff can choose `academic session -> module` because they are querying cohorts
- Student should choose `their session enrolment progression -> optional module` because they are querying personal progression

That solves the edge case neatly:
- Modular units: session filter narrows the cohort, module refines the units
- Non-modular certification units: session filter still works, module is ignored or unavailable
- No fake exclusion of year-long units just because `modules_taught` is null

My recommendation:
- For `student transcript`: use `session enrolment` as the main selector
- For `student marksheet`: also prefer `session enrolment`, with optional module
- Keep `year_of_study` only as derived display text from the selected session enrolment, not the driving filter

If we implement this, backend logic should shift from:
- `where units.year_of_study = X`
- optional `where units.modules_taught = Y`

to:
- start from selected `academic_session_enrolment_id`
- fetch that enrolment’s registered units from `student_unit_registrations`
- optionally refine by `modules_taught`
- compute marks/grade only from published marks for those registrations

That is more correct than using year alone.
